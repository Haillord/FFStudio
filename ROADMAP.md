# FFStudio Roadmap

Этот документ не про "все хорошие идеи сразу", а про реальный порядок работ.
Цель: довести приложение до состояния, где оно:

- предсказуемо работает в основных сценариях,
- не обманывает пользователя декоративными настройками,
- дает понятные ошибки,
- безопасно обращается с внешними бинарями и локальными процессами,
- собирается и проверяется автоматически.

---

## Главный принцип

Сначала доводим до надежного состояния то, что уже есть.
Новые функции добавляем только после того, как:

- базовые сценарии не ломаются,
- ошибки читаемы,
- отмена реально работает,
- сборка проходит автоматически,
- мы понимаем, что каждая настройка в UI действительно влияет на результат.

Иначе проект будет расширяться быстрее, чем стабилизироваться.

---

## Текущее положение проекта

Сейчас у проекта уже есть хороший функциональный объем:

- FFmpeg-конвертация,
- voice/TTS сценарии,
- image generation через ComfyUI,
- внешние runtime-интеграции,
- отдельные вкладки с уже неплохим UX.

Но есть системные риски:

- `src-tauri/src/main.rs` слишком большой и совмещает слишком много ответственности,
- ошибки и отмена задач реализованы неравномерно,
- часть UX уже хорошая, но не унифицирована,
- есть внешние скачивания и запуск сторонних бинарей без достаточного уровня защиты,
- нет CI как обязательного quality gate,
- roadmap в старом виде описывает темы, но не помогает понять "что делать следующим шагом".

Поэтому ниже roadmap переписан в правильном порядке: сначала стабилизация, потом архитектура, потом расширение продукта.

---

## Этап 1. Стабилизация существующего продукта

Это главный этап. Пока он не завершен, не нужно активно расширять продукт новыми крупными функциями.

### 1.1 Единый UX ошибок

Что делать:

- убрать `alert` и `confirm` как основной путь ошибок и подтверждений,
- сделать единый слой отображения ошибок:
  - короткое понятное сообщение,
  - кнопка копирования,
  - технические детали при необходимости,
- унифицировать формат ошибок между frontend и Tauri backend,
- ввести правило: любая ошибка должна быть полезной и пользователю, и разработчику.

Почему это важно:

- сейчас часть ошибок уже отображается хорошо, а часть еще уходит в `alert`,
- это делает приложение ощущаемо менее зрелым,
- без единого error UX тяжело отлаживать реальные кейсы пользователей.

Definition of Done:

- в ключевых сценариях нет `alert` как основного UX для ошибок,
- ошибки выглядят единообразно,
- ошибки можно скопировать,
- backend отдает достаточно контекста для диагностики.

### 1.2 Реальная отмена задач

Что делать:

- проверить все долгие операции:
  - FFmpeg,
  - Voice/TTS,
  - ComfyUI / image generation,
  - новые внешние runtime вроде `VCClient`,
- убедиться, что кнопка "Отмена" реально завершает процесс, а не только сбрасывает UI,
- привести lifecycle процессов к одному паттерну:
  - старт,
  - progress,
  - cancel,
  - cleanup,
  - финальный статус.

Почему это важно:

- надежная отмена важнее, чем новые фичи,
- пользователь должен понимать, что длинный процесс управляем.

Definition of Done:

- каждая долгая операция имеет настоящий cancel path,
- после отмены не остаются висящие процессы,
- UI после отмены возвращается в корректное состояние.

### 1.3 Аудит соответствия UI и реального поведения

Что делать:

- пройти по всем ключевым вкладкам и составить таблицу:
  - контрол в UI,
  - куда он маппится,
  - реально ли влияет на backend / команду,
- убрать или доработать декоративные параметры,
- проверить, что настройки из `Settings` действительно используются.

Приоритет проверки:

1. `Audio`
2. `Voice`
3. `Image`
4. `Realtime Voice`
5. `Video`
6. остальные вкладки

Почему это важно:

- главная потеря доверия к приложению происходит там, где UI обещает одно, а результат не меняется,
- перед расширением функционала нужно сначала закрыть эти разрывы.

Definition of Done:

- нет критичных мертвых контролов,
- все важные настройки реально влияют на результат,
- лишние или неподдерживаемые настройки либо удалены, либо честно помечены.

### 1.4 Базовая автоматическая проверка

Что делать:

- добавить CI:
  - frontend build,
  - `cargo check`,
  - при возможности smoke-проверки,
- добавить хотя бы минимальные тесты на чистые функции:
  - сборка ffmpeg args,
  - генерация ComfyUI workflow,
  - небольшие utility-функции без UI.

Почему это важно:

- сейчас проект уже достаточно большой, чтобы ручная проверка перестала быть достаточной,
- CI нужен не "когда-нибудь потом", а до активного роста функционала.

Definition of Done:

- любой PR проходит автоматическую базовую проверку,
- есть хотя бы минимальная защита от регрессий в самых важных вычисляемых частях.

## Этап 2. Security hardening

Этот этап обязателен до стабильного публичного релиза, но его лучше делать после базовой стабилизации UX и поведения.

### 2.1 Ограничение Tauri capability и shell scope

Что делать:

- пересмотреть доступы файловой системы,
- убрать избыточные глобальные разрешения,
- ограничить shell-команды до необходимого минимума,
- по возможности перейти от широких разрешений к явным сценариям.

### 2.2 CSP и защита webview-слоя

Что делать:

- включить CSP,
- убрать небезопасные конфигурации вида `csp: null`,
- проверить, не ломает ли это текущие вкладки и внешние интеграции.

### 2.3 Валидация внешних загрузок

Что делать:

- для всех скачиваемых архивов и бинарей добавить проверку целостности,
- фиксировать версию, URL и hash,
- не считать установку успешной, если архив не прошел проверку.

Почему этап 2 идет после этапа 1:

- security hardening обязателен,
- но если сначала не стабилизировать UX и поведение, команда будет постоянно тушить сразу все типы проблем вперемешку.

Definition of Done:

- capabilities ограничены,
- CSP включен,
- внешние загрузки валидируются по checksum,
- shell/scopes не шире необходимого.

---

## Этап 3. Архитектурная уборка

Делать только после этапов 1 и 2.

### 3.1 Разделить `src-tauri/src/main.rs`

Цель:

- убрать монолитный backend-файл,
- сделать проект понятнее и безопаснее для изменений.

Предлагаемая структура:

- `src-tauri/src/core/`
  - logging
  - error helpers
  - process management
- `src-tauri/src/ffmpeg/`
- `src-tauri/src/voice/`
- `src-tauri/src/sd/`
- `src-tauri/src/installer/`
- `src-tauri/src/system/`

Что важно:

- не делать "рефактор ради красоты",
- выносить только после того, как понятны реальные границы модулей.

Definition of Done:

- `main.rs` становится входной точкой и маршрутизацией команд,
- основная логика вынесена по доменам,
- новые фичи больше не требуют бесконтрольно раздувать один файл.

---

## Этап 4. Выравнивание UX по всему приложению

После стабилизации и архитектурной уборки.

Что делать:

- унифицировать паттерны:
  - прогресс,
  - ошибки,
  - cancel,
  - success state,
  - copy/open actions,
- централизовать drag&drop listeners,
- уменьшить дублирование UI-логики между вкладками,
- оптимизировать работу с большими base64 и тяжелыми preview-сценариями.

Definition of Done:

- вкладки ощущаются как части одного продукта, а не как набор отдельных мини-приложений,
- одинаковые сценарии работают одинаково во всех разделах.

---

## Этап 5. Расширение функционала image generation

Только после того, как база стала надежной.

### 5.1 Улучшение SD 1.5 workflow

Что делать:

- отдельный `VAELoader`,
- поддержка нескольких LoRA,
- опциональный HiRes Fix,
- более удачный sampler по умолчанию,
- рекомендации параметров под sampler,
- честный UI только для тех параметров, которые реально поддерживаются workflow.

### 5.2 Отдельная вкладка Flux

Что делать:

- простая вкладка без перегруженного UI,
- отдельный builder workflow,
- минимальный, но удобный набор параметров.

Важно:

- не повторить старую ошибку "сначала UI, потом думаем, как он реально работает",
- сначала backend/workflow, потом интерфейс.

Definition of Done:

- новые image-сценарии реально полезны,
- UI не содержит мертвых контролов,
- сценарии проходят базовую автоматическую проверку.

---

## Этап 6. Release-инфраструктура

Когда продукт уже технически стабилен.

Что делать:

- ввести release checklist,
- оформить changelog,
- подключить updater только после security hardening,
- определить, что считается supported environment:
  - Windows версии,
  - FFmpeg,
  - Python,
  - GPU / CPU сценарии,
  - внешние runtime.

Definition of Done:

- релиз можно выпускать повторяемо,
- есть понятный список проверок перед выпуском,
- known issues и ограничения зафиксированы явно.

---

## Этап 7. Полировка продукта

Это уже не блокирует релиз, но повышает качество продукта.

### 7.1 Локализация и доступность

- вынести строки,
- внедрить RU/EN,
- сделать базовые a11y улучшения.

### 7.2 Onboarding и help

- first-run проверка зависимостей,
- troubleshooting раздел,
- рекомендации по runtime и моделям.

### 7.3 Документация

- актуальный README,
- инструкции по установке,
- документация по моделям, семплерам и runtime.

---

## Что делаем следующим

Если действовать правильно, следующий рабочий порядок такой:

1. Единый error UX вместо `alert/confirm`.
2. Реальная отмена задач во всех длинных сценариях.
3. Аудит `UI -> backend` по ключевым вкладкам.
4. CI: build + `cargo check` + минимальные тесты.
5. Только потом security hardening.
6. Потом модульный рефактор backend.

Именно так roadmap должен работать на практике.

---

## Минимальный критерий "готово к стабильному релизу"

Считать проект готовым к стабильному релизу можно только если:

- завершен этап 1,
- завершен этап 2,
- нет критичных багов в сценариях:
  - Video,
  - Audio,
  - Voice,
  - Realtime Voice,
  - Image generation,
- есть CI,
- есть понятный release checklist,
- внешние загрузки и запуск сторонних runtime защищены на приемлемом уровне.
